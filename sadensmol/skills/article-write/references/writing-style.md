# sadensmol.com Writing Style Guide

## Voice & Tone

- First person, conversational - write as if talking to a fellow developer
- Direct and opinionated - don't hedge, state your views clearly
- Honest about limitations and trade-offs - acknowledge "bad sides" openly
- Self-deprecating humor when appropriate
- Use casual language: "ppl", "isn't much", contractions freely
- Enthusiastic about things you like - use "!" naturally
- Critical but constructive - explain WHY something is a problem

## Structure Patterns

### Problem-focused articles (most common)
1. Hook/intro (1-3 sentences, jump right in)
2. "Why is it a problem?" section
3. "What are our options?" or solution section
4. Code examples with explanation
5. Summary/Conclusion (short, actionable)
6. Optional: "PS:" inviting feedback or discussion

### Comparison/experience articles
1. Personal background/context
2. Why you made the switch/choice
3. "Good sides" section with subsections
4. "Bad sides" section with subsections
5. Conclusion with honest assessment
6. Resources section with links

### Series articles (e.g. "Things I Hate in Go")
1. Brief intro connecting to series theme
2. The specific issue explained
3. Why it matters (real-world impact)
4. Comparison with other languages if relevant
5. Short conclusion or call for discussion

## Formatting Rules

- Use `##` for main sections, `###` for subsections
- Short paragraphs (1-4 sentences max)
- Code blocks with language tags (```go, ```bash)
- Bare URLs for resources (just paste the URL on its own line)
- Images referenced as `![alt text](/images/posts/filename.ext)`
- Keep articles concise: 50-250 lines typical
- No bullet-point walls - prefer prose with occasional lists
- Bold for emphasis on key terms: **currency units**, **pass floats as strings**

## Example Phrases (to match voice)

- "We all know..."
- "The main question and problem here..."
- "The answer is simple..."
- "But you can ask..."
- "Let's try to figure it out!"
- "Simple! And I quite like it!"
- "This is the next problem..."
- "Thanks for reading!"
- "Let me know your thoughts on this"
- "But it's a theme for another article!"

## What NOT to do

- Don't use corporate/formal language
- Don't write long introductions
- Don't use "In this article, we will..." preambles
- Don't over-explain obvious things to experienced developers
- Don't use AI-sounding phrases like "Let's dive in", "In conclusion", "It's worth noting"
- Don't add unnecessary disclaimers or caveats
- Don't write walls of text without code examples
